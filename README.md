# OmniSLA (Decentralized SLA Heartbeat Monitor)

OmniSLA is a decentralized, trustless Service Level Agreement (SLA) monitoring and enforcement framework built as an Intelligent Contract on GenLayer. It programmatically enforces service quality agreements between a Service Provider and a Client by locking funds in escrow and evaluating endpoints using consensus-verified web crawls and LLM evaluations.

## Equivalence Principle & Consensus Design

Bringing off-chain data (such as web service status pages) and non-deterministic logic (such as natural language LLM evaluations) onto a blockchain is traditionally difficult due to the oracle problem. If validators get different web responses or LLM answers, the network cannot reach consensus, leading to chain splits.

GenLayer solves this problem using the Equivalence Principle. OmniSLA utilizes this design to evaluate SLAs trustlessly:

1.  Isolated Execution (Leader Node):
    *   When check_sla() is triggered, the elected Leader node executes the non-deterministic execute_check() block in isolation.
    *   The Leader crawls the target web URL and runs the LLM model to analyze the content against the SLA rule.
    *   The Leader packages these findings into a structured JSON verdict.

2.  Consensus Validation (Validator Nodes):
    *   Validator nodes execute validate_check(leader_result) independently.
    *   Each validator performs their own isolated web crawl and LLM call.
    *   To reach consensus, validators must agree on the core parameters of the check.

3.  Deterministic Comparison:
    *   To account for minor differences in natural language explanations generated by LLMs, the validation logic compares only the deterministic fields of the verdict:
        *   condition_satisfied (whether the service is compliant)
        *   failure_category (Network, Server, Content, Semantic, None)
        *   severity (None, Low, Medium, High)
    *   This comparison guarantees that as long as validators agree on the status, category, and severity of the service, consensus is achieved. This makes off-chain monitoring resilient, decentralized, and immune to single-node manipulation.

## Key Features

*   Trustless Escrow: The client deposits the premium and the provider deposits the collateral. The contract activates automatically only when both parties have fully funded their shares.
*   Spam Protection: Prevents repeated instant status checks using a configurable check_interval_seconds cooldown.
*   Consecutive Failure Policy: Protects providers from temporary, transient network glitches by requiring consecutive failed checks before trigger slashing.
*   Structured Verdict Logging: Logs detailed failure categories (Network, Server, Content, Semantic, None) and severity levels rather than simple binary outcomes.
*   Prompt Injection Defense: The semantic LLM prompt is isolated with system instructions to treat webpage content strictly as passive data, blocking injection overrides.

## Example Use Cases

OmniSLA is a modular contract primitive designed to be easily forked, configured, or integrated into a variety of decentralized infrastructure services:

1.  Autonomous Web Service SLA Monitor (Decentralized Heartbeat)
    *   SaaS providers and clients lock collateral and premiums. The contract evaluates the status page or health check endpoint. If the site reports errors or remains unreachable for successive heartbeat checks, the provider is automatically slashed.

2.  Oracle and Data Feed Compliance
    *   Off-chain data nodes lock collateral. The contract periodically queries the oracle feeds to ensure they contain correct, freshly updated data structures. Slashes are triggered if data remains stale or contains malformed payloads.

3.  Decentralized CDN and Hosting Availability Escrow
    *   Decentralized web hosters lock collateral to serve websites. Heartbeat monitors check if index files contain specific hash values or script patterns to guarantee CDN availability and content integrity.

4.  AI and LLM API Quality Assurance Escrow
    *   Third-party AI API providers lock collateral. Clients lock premiums. Heartbeat checks verify that the LLM response complies with strict output formats (such as compliant JSON payloads) or does not exceed maximum response latencies.

5.  Decentralized Job Execution Verification
    *   Off-chain computation workers check in by updating decentralized task endpoints. OmniSLA monitors the status page, releasing escrow payouts upon successful completion and slashing on task timeouts.

## Getting Started

### 1. Set Up the Python Environment
Create a virtual environment and install the required dependencies (requires Python >= 3.12):

```bash
python3.12 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 2. Lint the Contract
Use the GenVM static analyzer to verify that the contract complies with all GenLayer security and non-deterministic rule constraints:

```bash
genvm-lint contracts/OmniSLA.py
```

### 3. Run the Test Suite
Run the fast, in-memory direct mode tests containing mocks for web requests and LLM evaluations:

```bash
PYTHONPATH=. pytest tests/direct/
```

## Contract API Details

### Escrow Setup
*   `deposit_collateral()`: Payable function for the provider to fund their collateral.
*   `deposit_premium()`: Payable function for the client to fund the service premium.
*   `refund()`: Allows either party to withdraw their deposits before the SLA is activated.

### Heartbeat & Payouts
*   `check_sla()`: Triggers the web search and validation checks. Can be called by anyone or cron triggers.
*   `close_sla()`: Allows the provider to collect their payout once the SLA end time has expired.
